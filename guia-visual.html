JSON Web Tokens · Stateless · bcrypt

POST /login con form-data (OAuth2 standard) El frontend envía las credenciales al backend

6
Cada request lleva el token en el header Authorization: Bearer eyJ...
7
Backend decodifica → extrae user_id → sabe quién es Sin consultar la DB · Stateless · Cualquier instancia puede validar
🔍 Estructura de un JWT
eyJhbGciOiJIUzI1NiJ9
HEADER (algoritmo)
.
eyJzdWIiOiIxIn0
PAYLOAD (user ID)
.
firma_hash_con_SECRET_KEY
SIGNATURE (integridad)

El header y payload son solo Base64 (cualquiera los lee). La firma garantiza que nadie los modificó, porque solo el servidor tiene el SECRET_KEY.

✅ Stateless (JWT)

El servidor NO guarda nada en memoria. Cualquier instancia puede validar cualquier token. Escalable horizontalmente.

🔒 bcrypt

Lento por diseño (cost factor). Si roban la DB, tardaría años en fuerza bruta. Nunca texto plano.

Python en vez de SQL crudo · Seguridad · Portabilidad
❌ Sin ORM (SQL crudo)

✅ Con SQLAlchemy ORM

Ventajas del ORM
🛡️
Seguridad — Previene SQL Injection automáticamente (los valores se escapan)

🔗
Relaciones — Las FK se manejan como atributos Python: project.tasks, task.comments

🚨 Preguntas Frecuentes — Con Respuestas
Las preguntas más comunes y cómo responderlas
❓ ¿Por qué FastAPI y no Django o Flask?
FastAPI es moderno, asíncrono y tiene validación automática con Pydantic integrada. Genera documentación Swagger en /docs automáticamente. Flask requiere más configuración manual. Django es monolítico y difícil de separar en capas.
❓ ¿Por qué MariaDB y no PostgreSQL o MongoDB?
El modelo tiene relaciones claras y bien definidas → base de datos relacional es la elección natural. MariaDB es compatible con MySQL, liviana, perfecto para Docker. PostgreSQL también hubiera sido válido.
❓ ¿Qué es la arquitectura en capas y por qué la usaste?
Patrón donde cada capa tiene una responsabilidad única: controller → HTTP, service → reglas, repository → datos. Si mañana cambio de SQL a NoSQL, solo toco el repository. El resto del código no se entera.
❓ ¿Qué es un JWT y cómo funciona?
Token firmado en 3 partes: header · payload · signature. El payload tiene el user ID ("sub"). La firma usa el SECRET_KEY. El cliente lo guarda y lo envía en cada request. El servidor lo decodifica y verifica la firma sin consultar la DB.
❓ ¿Por qué guardás el token en localStorage?
Para un challenge de alcance acotado es práctico. En producción real lo ideal son httpOnly cookies (no accesibles desde JS, protección contra XSS). Para evaluación, localStorage simplifica el flujo y cumple el requisito.
❓ ¿Qué es el healthcheck en Docker?
depends_on solo garantiza que el contenedor arrancó, no que MariaDB esté lista. La DB tarda ~20s en inicializar. Sin healthcheck, el backend crashea. Con condition: service_healthy, espera a que MariaDB pase su propio test.
❓ ¿Qué mejorarías si tuvieras más tiempo?
1) httpOnly cookies + refresh tokens · 2) Tests unitarios con pytest · 3) Separar configs dev/prod · 4) Paginación en listados · 5) Endpoint PATCH para editar status de tarea desde la UI.
❓ ¿Cómo escalarías esto?
La arquitectura stateless facilita escalar horizontalmente: múltiples instancias del backend porque el estado está en la DB. Para tiempo real, agregaría WebSockets en un nuevo controller sin tocar el resto.

📚 Glosario — Definiciones en Tu Sistema
Qué significa cada término y dónde aparece exactamente en este proyecto
Arquitectura en capas Backend

Patrón que divide el sistema en capas con una sola responsabilidad cada una. Si algo cambia (la DB, el protocolo HTTP, las reglas del negocio), solo tocás una capa. El resto no se entera.

📍 En tu sistema controllers/ → recibe la HTTP request y delega
services/ → aplica las reglas de negocio
repositories/ → ejecuta queries en MariaDB
Single Responsibility Backend

Principio SOLID: cada clase tiene una sola razón para cambiar. Si una clase mezcla HTTP + lógica + DB, cualquier cambio puede romper cosas no relacionadas.

📍 En tu sistema task_controller.py → solo rutas HTTP, nada más
task_service.py → solo reglas de negocio
task_repository.py → solo consultas a la DB
Stateless / Stateful Backend

Stateless: el servidor no recuerda nada entre requests. Cada una lleva todo lo necesario para ser procesada. Ideal para escalar horizontalmente. Stateful: el servidor mantiene sesiones en memoria (lo opuesto).

📍 En tu sistema Tu FastAPI es STATELESS: verifica el JWT de la cookie
en cada request, sin tablas de sesiones en la DB.
El "estado" vive en el token, no en el servidor.
JWT / Bearer Token Seguridad

JWT (JSON Web Token): token firmado de 3 partes — header (algoritmo), payload (datos: user ID + expiración), signature (firmado con SECRET_KEY). Bearer = "quien lo porta tiene permiso". El servidor lo verifica sin consultar la DB.

📍 En tu sistema auth.py → create_access_token({"sub": str(user.id)})
auth.py → get_current_user_id() lee la httpOnly cookie
El frontend envía la cookie automáticamente (credentials:"include")
OAuth2 Seguridad

Estándar de autorización que define distintos flujos para obtener tokens de acceso. FastAPI integra el flujo "Password" por conveniencia: el cliente envía usuario + contraseña como form-data.

📍 En tu sistema POST /login recibe form-data (username, password)
FastAPI usa OAuth2PasswordRequestForm para parsearlo
Sigue el estándar OAuth2 Resource Owner Password Flow
ORM / Migration Base de datos

ORM (Object-Relational Mapper): traduce clases Python en tablas SQL y viceversa. Escribís Python puro, el ORM genera el SQL. Migration: script versionado que aplica o revierte cambios al esquema.

📍 En tu sistema models.py → class Task(Base) define columnas en Python
SQLAlchemy ORM genera el SQL correspondiente
Base.metadata.create_all() crea las tablas al iniciar
Foreign Key / Cascade Base de datos

Foreign Key: columna que referencia el ID de otra tabla, estableciendo una relación entre entidades. Cascade: efecto en cadena — si se elimina el registro padre, se eliminan automáticamente sus hijos.

📍 En tu sistema tasks.project_id → FK a projects.id (obligatoria)
tasks.assignee_id → FK a users.id (opcional / nullable)
Borrar un proyecto → sus tareas se borran en cascada
Borrar una tarea → sus comentarios se borran en cascada
Docker / Imagen / Volumen Infraestructura

Imagen: plantilla inmutable con todo el entorno. Contenedor: instancia en ejecución de una imagen. Volumen: almacenamiento persistente externo al contenedor — los datos sobreviven aunque el contenedor se destruya.

📍 En tu sistema backend/Dockerfile → imagen Python 3.11-slim + FastAPI
frontend/Dockerfile → imagen Node 20-alpine (multi-stage)
docker-compose.yml → volumen db_data persiste MariaDB
sin él, los datos se pierden al hacer docker compose down
REST / Endpoint Backend

REST: estilo donde los recursos se identifican por URL y se manipulan con verbos HTTP (GET leer, POST crear, PUT/PATCH actualizar, DELETE eliminar). Endpoint: combinación verbo + ruta que expone una operación.

📍 En tu sistema GET /projects/my-projects → listar mis proyectos
POST /tasks/ → crear nueva tarea
GET /comments/task/{id} → comentarios de una tarea
POST /login · POST /logout → autenticarse / cerrar sesión
HTTP 200 / 201 / 401 / 403 / 404 / 422 Backend

Códigos que el servidor envía para indicar el resultado. 2xx = éxito. 4xx = error del cliente. Saber cuál usar en cada situación es señal de buena práctica REST.

📍 En tu sistema 200 OK → GET exitoso (listar proyectos/tareas)
201 Created → POST exitoso (nueva tarea o proyecto)
401 Unauthorized → cookie ausente o JWT inválido/expirado
403 Forbidden → autenticado pero sin permiso (no es dueño)
404 Not Found → proyecto o tarea no existe en la DB
422 Unprocessable → body inválido (Pydantic lo dispara solo)
Pydantic / Schema Backend

Pydantic: librería Python que valida datos usando type hints. Schema: clase que define la forma esperada de los datos de entrada (request body) o salida (response). Si el dato no cumple → HTTP 422 automático.

📍 En tu sistema schemas.py → TaskCreate: title(str), project_id(int)...
Si enviás project_id:"abc" → HTTP 422 sin código extra
TaskResponse define exactamente qué devuelve la API
UserResponse excluye hashed_password de la respuesta
bcrypt / Hash Seguridad

Hash: función de una sola vía — convierte texto en una cadena fija irreversible. No se puede "deshacer". bcrypt: algoritmo de hashing con cost factor configurable, diseñado para ser lento y resistir fuerza bruta.

📍 En tu sistema auth.py → get_password_hash(password) al registrarse
auth.py → verify_password(plain, hashed) al hacer login
users.hashed_password NUNCA guarda texto plano en la DB
Route Guard Frontend

Mecanismo de protección de rutas del lado del cliente. Antes de renderizar una página privada, verifica si el usuario está autenticado. Si no lo está, redirige inmediatamente al login.

📍 En tu sistema En page.tsx, /project/[id] y /task/[id]:
fetch("http://localhost:8000/users/me", {credentials:"include"})
si status === 401 → router.push("/login")
La httpOnly cookie es el "pase" — el navegador la envía sola
Multi-stage build Infraestructura

Técnica de Dockerfile que usa múltiples etapas: una para compilar con todas las herramientas de desarrollo, y otra final solo con el artefacto compilado. La imagen de producción es mucho más pequeña y no expone el código fuente.

📍 En tu sistema — frontend/Dockerfile FROM node:20-alpine AS builder
RUN npm ci && npm run build ← compila TypeScript
FROM node:20-alpine AS runner
COPY --from=builder /app/.next ./.next ← solo lo compilado
Healthcheck Infraestructura

Test periódico que Docker ejecuta para saber si el servicio interno está realmente operativo, no solo si el proceso existe. Permite que otros servicios esperen confirmación de que el vecino está listo antes de arrancar.

📍 En tu sistema — docker-compose.yml db → healthcheck: mysqladmin ping -h localhost
backend → depends_on: db (condition: service_healthy)
sin esto el backend tira "Connection refused" porque
MariaDB tarda ~20s en inicializar sus archivos internos
SQL Injection Seguridad

Ataque donde el usuario envía código SQL malicioso como input para manipular consultas. Si el servidor concatena el input directamente al SQL, el atacante puede leer, modificar o destruir datos de toda la base de datos.

📍 En tu sistema — ¿Estás protegido? ✅ SÍ SQLAlchemy ORM usa parámetros enlazados (bind params)
db.query(Task).filter(Task.id == task_id) es seguro
Los valores nunca se concatenan como strings al SQL
El ORM los escapa automáticamente antes de ejecutar
